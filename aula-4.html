Teste de Integração

A integração do jogo é o momento em que o que foi produzido, é colocado no projeto base dentro da engine, para um processo muito importante no final da etapa de produção. Nele, primeiramente são realizados os testes para implementação dos primeiros elementos para o jogo, montando tudo para formar uma versão voltada para testar cada um desses elementos.

Nesse teste, não há apenas a ação do programador, mas também de toda a equipe, inclusive produtor do jogo. É um trabalho em conjunto para que seja verificado e testado cada elemento que precisa estar presente no jogo para sua publicação. Por enquanto, serão testados alguns objetos de cada aspecto, isto é, elementos da narrativa, estética, dinâmica, mecânica, áudio etc.

As etapas finais da produção são momentos em que o produtor do jogo pode contribuir bastante, por meio de análises e sinalizações, ele precisa garantir que o planejado seja cumprido e integrado ao jogo, visando a sua publicação.

É importante saber selecionar os elementos para essa etapa, primeiramente devem ser escolhidos aqueles que mais possuem interação no jogo, ou são de extrema importância em momentos chaves, como botões muito usados da interface ou o boss (chefe) da primeira fase, por exemplo. Depois, os demais elementos, para que tudo seja aprovado.

Nesse momento, realizar uma documentação com clareza é importante para deixar todos os envolvidos, nessa etapa, cientes do que está sendo testado, como no exemplo da tabela a seguir:

Interface Personagens Inimigos Cenário
Botão play Protagonista Inimigo terrestre Porta de saída
Barra de vida do jogador Mentor Inimigo voador Plataforma móvel
Texto de pontuação Princesa Boss Alavanca

Recomenda-se a escolha de poucos elementos, mas que sejam essenciais para obter um feedback preciso do funcionamento de cada área do jogo. A pouca quantidade contribui para agilizar o processo, para ficar melhor sua compreensão, vamos usar o caso de teste de um inimigo no jogo como exemplo, não é necessário nesse momento colocar todos os inimigos desse mesmo tipo, apenas um é o suficiente.

Uma forma de resumir o que é feito nessa etapa é como o teste de cada elemento que será integrado no jogo, checando se:

  • Sua mecânica está funcionando como deveria;
  • Se a animação está fluída;
  • Se os conceitos, como o do personagem principal, está em sintonia com o resto;
  • Se os efeitos sonoros estão sendo tocados corretamente etc…

    • Essas, entre outras características que o game designer ou o produtor achar necessário, podem ser testadas, no caso do exemplo que usamos, com apenas um inimigo, pois uma vez que a instância que está sendo testada funciona como deveria, logicamente, todas as outras desse mesmo inimigo também estarão.

    Um exemplo de teste de integração com elemento narrativo, seria como a caixa de diálogo é aberta enquanto o jogador interage com algum personagem que lhe informa algo da história. Assim, digamos que a programação dessa parte é feita para, independentemente do personagem que o jogador esteja interagindo, a caixa de diálogo seja aberta e apareça em escrito o que o personagem diz.

    Então, para testar a integração no caso desse exemplo, basta usar apenas um personagem, pois todos os outros que compartilham da mesma programação, estarão funcionando. De forma semelhante funciona a programação para as animações e comportamentos dos inimigos do jogo.

    Após os últimos elementos escolhidos serem implementados, além do teste da qualidade de função (se funcionam exatamente como deveriam), também é feita localização de possíveis bugs ou brechas na mecânica, o que pode não ser necessariamente um bug, mas não estão de acordo com o proposto pelo game designer.

    Alguns jogadores mais astutos conseguem tirar proveito de brechas nas mecânicas dos jogos para terem certa vantagem, essas brechas são mais conhecidas como Glitchs, que são falhas na programação, mas que não causam a quebra do jogo como os Bugs, apenas respostas inesperadas da mecânica do jogo.

    Controle de versões

    Ao começar a integrar os primeiros elementos é comum a criação de várias versões do projeto. Ao criar essas versões se obtém um controle melhor do processo de integração, além do que está sendo testado.

    O objetivo básico de um sistema de controle de versão é salvar todo o histórico da produção do jogo, desde a primeira até a versão mais recente. No desenvolvimento de jogos é muito importante ter uma espécie de backup de versões, pois pode ser que elementos possam ser retirados, adicionados ou sofrer uma grande alteração e, algumas vezes, esses elementos influenciam, de forma funcional, em muita coisa dentro jogo.

    Por isso, mesmo antes de haver a integração das versões, recomenda-se a elaboração de um sistema de controle de versões específico para o projeto de jogo. Acompanhe a seguir alguns dos principais benefícios da utilização de um sistema de controle de versão para controlar alterações feitas durante essa etapa da produção do jogo:

  • Possibilidade de backups: Facilidade de manusear o histórico de desenvolvimento, além de recuperar versões anteriores e mais estáveis. A maioria das implementações permite examinar.
  • Trabalho em equipe: As equipes podem trabalhar juntas, mesmo em áreas de desenvolvimento diferentes, testando a mesma versão, reduzindo a ocorrência de problemas causados por conflitos de versão. É possível que a solução ainda inclua controle de acesso para cada área.
  • Análises em paralelo: Possibilita que todas as áreas de desenvolvimento, possam analisar a mesma versão do jogo ao mesmo tempo de forma segura, mas com foco em suas respectivas áreas de atuação, isto é, a equipe de arte pode verificar tudo que precisa no ritmo necessário sem atrapalhar a análise das outras áreas.
  • Identificação: Utilizando uma nomenclatura própria e fácil de ser identificada, cada versão precisa ser localizada de forma prática e clara, geralmente são usados números, mas podem ser usadas letras quando em conjunto, por exemplo: “nome do jogo” v1.0, “nome do jogo” v2.1 ou “nome do jogo” v1a, “nome do jogo” v1.2b. Ficando a critério do programador quando esse exportar a nova versão da engine.

    • Recomenda-se que o envio de novas versões, que é feito a critério da necessidade, seja enviada sempre que a mesma esteja no mínimo estável, para minimizar conflitos de versão, facilitar o desfazer de alterações e controlar o histórico. Entregando correções ou uma atualização relevante que funcione corretamente. Não é recomendado enviar quando a versão pode apresentar um problema para o desenvolvimento de outro colaborador, como código não compilado ou uma falha que comprometa toda a execução.

    Existem modelos de sistemas de controle de versão já prontos como o CVS, Git, Subversion, Mercurial entre outros, que organizam arquivos de maneira lógica para criação, acesso controlado, atualização e exclusão. A automação e controle de versão estão intimamente ligados, bem como a integração e configuração contínua como os dados a serem entregues. A seguir, está um exemplo de um histórico de versões utilizando o sistema CVS.

    O modelo CVS emprega uma arquitetura cliente-servidor, na qual um servidor mantém as versões e o histórico atuais do projeto, o resto da equipe se conecta a esse servidor para adquirir uma cópia completa do projeto, trabalhar nele e retornar suas alterações.

    Mas, claro! Cabe ao produtor ou por meio de um consenso entre todos os membros da equipe, de utilizar um desses sistemas prontos, pois o que é realmente necessário e que contribui bastante para evitar problemas nessa fase final do processo de produção.

    Integração de versões

    Na etapa de integração de versões é onde a equipe de programação vai gerar uma nova versão atualizada, após os testes de integração, passará por novos testes, mas agora para saber como está sendo executada.

    É importantíssimo que o elaborador da versão nomeie as versões de forma clara e objetiva, pois ela será acessada por outras pessoas, além de que haverá outras versões.

    Dependendo da engine utilizada, o programador deverá exportar a nova versão para todas as plataformas em que o jogo pretende ser lançado, para colher informações de como está sendo executado nelas.

    Assim que uma nova versão é integrada, na pasta compartilhada pela equipe, recomenda-se que o autor dessa versão, avise a todos informando o nome completo dela, para que assim os membros que possam estar esperando-a para análise, poupem tempo e facilmente a identifique para continuar com seu trabalho.

    O programador precisa dominar esse processo de integração de versão, pois um ponto muito importante é que ele garanta a execução do jogo de forma otimizada, conseguindo um ótimo desempenho nas plataformas alvo.

    Isso é possível por meio de testes e balanceamento entre o que é implementado na versão e como é feita sua integração, pois mesmo que execute o processo para gerar uma nova versão sem falhas, ela pode apresentar problemas de desempenho durante a gameplay do jogo.

    Integração de arte e software

    É agora que todas as artes finais que foram produzidas para o jogo são integradas no projeto de forma definitiva, a equipe de programação precisa elaborar uma forma bem estruturada para implementar todas as artes no projeto de maneira ordenada, para que dessa forma tenho um melhor controle do que está sendo colocado e o que ainda falta.

    Essa etapa não possui muito segredo, mesmo sendo a equipe de programação os principais atuantes no momento, é importante que as outras áreas contribuam com uma fiscalização, muitas vezes sendo necessário sua participação para que por exemplo, a velocidade de uma animação que acabou de ser programada, que precisa reproduzir essa animação de acordo com o que foi planejado.

    Recomenda-se que seja utilizada uma tabela para controle do que está sendo implementado, nela, é listada todos os elementos que devem está no jogo, e a medida que cada um for sendo colocado, tal elemento deve ser marcado na lista informando que já foi integrado ao jogo, vale salientar que essa tabela precisa ficar disponível para toda a equipe.

    Esse processo é bastante semelhante ao que vimos no teste de integração, mas agora de forma bem mais ampla, pois contém todos os elementos visuais que devem estar no jogo, ou seja, todas as artes e modelos do cenário, dos personagens, objetos, interface, efeitos etc. Como na tabela a seguir, que como exemplo está com algumas células com preenchimento verde para indicar os elementos que foram integrados.

    Interface Personagens Inimigos Cenário
    Botão de menu Herói Inimigo terrestre Porta de saída
    Barra de energia Rei Inimigo voador Plataforma móvel
    Texto de Game Over Vendedor Boss Baú

    Simples não é? Mas é uma solução prática que pode evitar problemas, e dependendo do tamanho da equipe, recomenda-se ainda divisão do processo de integração, isto é, cada pessoa responsável para implementar determinados elementos, isso contribui para evitar que uma só pessoa fique sobrecarregada e possa esquecer ou cometer enganos na hora de preencher a tabela, podendo sinalizar que algum elemento foi colocado mas sem ter sido de fato, o que causaria retardo no desenvolvimento depois.

    Integração de componentes

    Um jogo digital pode apresentar uma variedade de componentes que interagem com o jogador, como música, efeitos sonoros e efeitos visuais são alguns exemplos. A maneira como as interações são realizadas tem uma influência direta na experiência de imersão do jogador.

    Cada componente integrado precisa ser testado depois, para caso apresente algum desacordo com o que foi planejado para o jogo, ou claro, alguma falha de execução, seja objetivamente registrado, para que então a equipe responsável pelo componente possa fazer as alterações necessárias.

    Uma boa recomendação é esperar que todos os componentes sejam testados para só depois mandar para que as alterações sejam realizadas, dessa maneira poupa-se tempo e reduz a quantidade de versões.

    Assim como as outras integrações, elaborar uma tabela para ter uma melhor administração de quais componentes são implementados nessa etapa, é muito importante. Tornando o processo mais rápido e contribuindo para evitar que faltem componentes ou a ocorrência de integração do que não deveria.

    Um bom exemplo de integração de componentes são os efeitos sonoros, geralmente colocados por último, quando o jogo em si está pronto, para isso, alguns programas já contam com recursos muito úteis que permitem que eles próprios sejam integrados na engine, como é o caso do FMOD e a Unity.

    Com isso, torna o processo de implementação dos efeitos muito mais prático, uma vez que é possível editar os sons que já estão dentro do projeto na engine, poupando o desenvolvedor da etapa de exportar o efeito novamente e depois importar para o jogo.

    Motivação

    Os jogos eletrônicos proporcionam entretenimento para pessoas de todas as idades. Sendo possível jogá-los com consoles, desktops e dispositivos móveis. A princípio, deveria ter sido projetado para o entretenimento para crianças e jovens, mas aos poucos se tornou um hobby para os adultos.

    Por causa da tremenda expansão da tecnologia de jogos e do realismo empregado em sua construção, o mercado de videogames agora produz bilionários, lidando com bastante dinheiro e criando uma legião de fãs.

    É importante verificar o processo de decisão dos jogadores, que inclui a busca para satisfazer um desejo ou necessidade, levando a uma pesquisa por mais informações, a avaliação de alternativas, a decisão de compra e o comportamento depois da compra, que contribuem para formar a opinião dele sobre o jogo.

    O primeiro passo para o jogador adquirir um jogo, é quando ele desperta o desejo por ele ou acha que precisa ter aquele jogo, muitas vezes ele tem uma concepção de que jogar uma ou duas vezes não é o bastante, ele precisa tê-lo pra jogar sempre que quiser.

    Mas para despertar esse desejo, o jogo precisa gerar motivação, que de acordo com o Dicio, Dicionário Online de Português, é o ato ou efeito de motivar, de despertar o interesse por algo; a reunião das razões pelas quais alguém age de certa forma e que dá origem a uma ação consciente.

    A geração de motivação é muito importante nos jogos, pois não só garante que o jogador irá comprar o jogo, como também é capaz de convencê-lo há, de forma espontânea, fazer propaganda ou recomendar o jogo para seus amigos.

    Nos jogos, ela está relacionada a necessidade de lazer que os seres humanos possuem, de ser capaz de distraí-lo quando usado como passatempo, por exemplo. Nesse sentido os jogos têm a capacidade de fazer com que o jogador possa se abstrair do seu cotidiano ao se envolver na história do jogo, por exemplo. A motivação de passatempo também influencia na compra e é semelhante ao da fuga da rotina, mas de maneira menos específica.

    Cabe ao desenvolvedor encontrar um meio de colher as informações necessárias do seu público, que depende do tipo de jogo que está produzindo. Por isso é muito importante conhecê-lo, para saber de fato os aspectos que o motivam a jogar, para então ser feita uma análise a fim de confirmar se o jogo atende a isso.

    Então por meio dos conhecimentos adquiridos com o estudo do público alvo, é possível focar no que vai conseguir satisfazer o jogador e fazê-lo gostar do jogo, nesse processo cabe ao game design ou produtor, analisar o jogo como um todo, para verificar se ele realmente atende ao que foi determinado como motivação para o jogador, geralmente com isso uma lista pode ser criada, apontando quais são essas motivações, como no exemplo a seguir:

  • Desbravar todas as áreas de um mundo cheio de magia;
  • Encontrar todos os segredos escondidos de uma civilização perdida;
  • Completar todos os desafios do jogo e acumular conquistas.

    • É importante salientar que as motivações dependem do estilo do jogo que está sendo produzido, podendo ser aspectos específicos desse jogo e não do gênero que faz parte. Essas motivações precisam ser facilmente atendidas durante o gameplay, de forma que o jogador não se esforce para notá-las, além de consegui-las de maneira clara e prática enquanto joga, para assim sentir-se jogando algo que gosta e se sentindo recompensado por ter, em determinado caso, comprado o jogo.

    Recapitulando

    Nesta unidade conhecemos as últimas etapas do processo de produção do jogo, que começa com o teste de integração dos principais elementos que estarão presentes nele, essa primeira etapa, tem o propósito de checar se o que foi produzido está funcionando como deveria, por isso utilizar somente uma representação de cada elemento é o mais recomendado para agilizar o processo

    Depois conhecemos como é importante ter o controle das versões, que traz vários benefícios, como backups, trabalho em equipe, ações em paralelo e uma melhor identificação de cada versão, tudo voltado para melhorar o trabalho de toda a equipe.

    Em seguida foi abordado a integração de uma nova versão e a implementação das artes finais do jogo, que inclui tudo que faz parte da estética do projeto, e nisso, pontuando mais uma vez a importância do uso de tabelas para ter um melhor controle do que está sendo integrado.

    A Integração de componentes, geralmente é a última parte do processo de integração, que é a etapa responsável pelo “acabamento” do jogo, esse componentes podem contribuem bastante para implementação de recursos que melhoram a imersão do jogo ou tornam a gameplay mais agradável.

    Por fim, foi falado a respeito da motivação que precisa ser gerada para que o jogador desperte o interesse pelo jogo. Mas para isso, é necessário conhecer o que o seu público prefere em um jogo do gênero escolhido, para assim garantir que o projeto seja concluído contendo as preferências do jogador e que vá conquistá-lo quando começar a jogar.
